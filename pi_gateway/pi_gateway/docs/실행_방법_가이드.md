# Pi Gateway 실행 방법 가이드

## 개요

Pi Gateway는 여러 컴포넌트로 구성되어 있으며, 각각의 역할에 따라 터미널을 분리하여 실행합니다.

**실행 방법**:
- **도커 실행 (권장)**: 메인 서버를 도커 컨테이너로 실행
- **로컬 실행**: 직접 Python으로 실행

---

## 🐳 도커 실행 방법 (권장)

### 1. 도커 이미지 빌드

```bash
cd /home/c203/pi_gateway
docker build -t pi-gateway:latest .
```

### 2. 도커 컴포즈로 실행 (권장)

```bash
cd /home/c203/pi_gateway
docker-compose up -d
```

**또는 직접 실행**:
```bash
docker run -d \
  --name pi-gateway \
  --network host \
  --device /dev/ttyUSB0:/dev/ttyUSB0 \
  -e ROS_DOMAIN_ID=1 \
  -e UART_ENABLED=1 \
  -e ROS_ENABLED=1 \
  -e WEBRTC_SIGNALING_ENABLED=1 \
  -e UART_PORT=/dev/ttyUSB0 \
  -e UART_BAUD=115200 \
  -e BE_WS_URL=wss://i14c203.p.ssafy.io/ws \
  -v $(pwd)/config:/app/config:ro \
  pi-gateway:latest
```

### 3. 로그 확인

```bash
# 실시간 로그 확인
docker logs -f pi-gateway

# 최근 100줄만 확인
docker logs --tail 100 pi-gateway
```

### 4. 컨테이너 관리

```bash
# 컨테이너 중지
docker-compose down
# 또는
docker stop pi-gateway

# 컨테이너 재시작
docker-compose restart
# 또는
docker restart pi-gateway

# 컨테이너 삭제
docker-compose down -v
# 또는
docker rm -f pi-gateway
```

### 5. 도커 실행 시 주의사항

- **네트워크 모드**: `host` 모드 사용 (ROS2 DDS 통신을 위해)
- **UART 디바이스**: `/dev/ttyUSB0` 디바이스를 컨테이너에 마운트 필요
- **ROS_DOMAIN_ID**: 호스트와 동일한 값으로 설정 (기본값: 1)

## 💻 로컬 실행 방법

### 터미널 구성 (3개)

### 터미널 1: Pi Gateway 메인 서버

**역할**:
- 웹 대시보드 서버 (FastAPI/WebSocket)
- ROS2 브릿지 (토픽 구독/발행)
- UART 통신 (STM32 하드웨어 제어)
- WebRTC 시그널링 클라이언트 (자동 시작) ⭐ **새로 추가됨**

**실행 명령**:
```bash
cd /home/c203/pi_gateway
source /opt/ros/humble/setup.bash
export ROS_DOMAIN_ID=1
export UART_ENABLED=1
export ROS_ENABLED=1
export WEBRTC_SIGNALING_ENABLED=1  # WebRTC 시그널링 클라이언트 활성화 (기본값: True)
python3 -m src.main
```

**또는 스크립트 사용**:
```bash
cd /home/c203/pi_gateway
bash scripts/run_gateway.sh
```

**확인할 로그**:
```
[INFO] WebRTC 시그널링 클라이언트 시작됨 (프론트엔드 버튼 클릭 감지 대기 중)
[INFO] 시그널링 서버 연결됨
[INFO] 스트리밍 시작 요청 토픽 구독 시작 (/sub/robot/stream/start)
[INFO] starting uvicorn on 0.0.0.0:8000 (MODE=ROS, ROS_ENABLED=True, UART_ENABLED=True, LOG_LEVEL=info)
```

**역할**:
- MQTT 브로커와 통신

**실행 명령**:

## WebRTC 스트리밍 동작 방식

### 자동 시작 방식 (권장)

**Pi Gateway 메인 서버가 자동으로 처리**:
1. 터미널 1에서 `python3 -m src.main` 실행
2. `webrtc_signaling_client.py`가 자동으로 시작됨 (별도 터미널 불필요)
3. 프론트엔드에서 버튼 클릭 → `/pub/robot/stream/start` 메시지 전송
4. Pi Gateway가 메시지 수신 → `robot_webrtc.py` 자동 시작
5. WebRTC 연결 완료

**장점**:
- 별도 터미널 불필요
- 자동 관리 (프로세스 시작/종료)
- 로그가 메인 서버와 함께 출력되어 관리 용이

### 수동 시작 방식 (선택사항)

만약 `robot_webrtc.py`를 수동으로 실행하고 싶다면:

**터미널 4 (선택사항)**: WebRTC 스트리밍
```bash
cd /home/c203/pi_gateway
source /opt/ros/humble/setup.bash
export ROS_DOMAIN_ID=1
export BE_WS_URL=wss://i14c203.p.ssafy.io/ws
export CAMERA_TOPIC=/front_cam/compressed
export ROBOT_ID=1
python3 scripts/robot_webrtc.py
```

**또는 스크립트 사용**:
```bash
cd /home/c203/pi_gateway
bash scripts/run_webrtc.sh
```

**주의**: 수동 실행 시 Pi Gateway의 자동 시작 기능과 충돌할 수 있으므로, 일반적으로는 자동 시작 방식을 권장합니다.

## 문제 해결

### 문제 1: WebRTC 시그널링 클라이언트가 시작되지 않음

**증상**: "WebRTC 시그널링 클라이언트 시작됨" 로그가 없음

**해결책**:
```bash
# 환경 변수 확인
export WEBRTC_SIGNALING_ENABLED=1
export ROS_ENABLED=1

# 의존성 확인
pip install websocket-client

# 재시작
python3 -m src.main
```

### 문제 2: robot_webrtc.py가 자동으로 시작되지 않음

**증상**: 프론트엔드 버튼 클릭 후에도 스트리밍이 시작되지 않음

**해결책**:
1. Pi Gateway 로그에서 "스트리밍 시작 요청 수신" 메시지 확인
2. 프론트엔드 콘솔에서 "스트리밍 시작 요청 전송" 로그 확인
3. 시그널링 서버 연결 상태 확인
4. 수동 실행으로 대체:
   ```bash
   # 터미널 4에서 수동 실행
   python3 scripts/robot_webrtc.py
   ```

### 문제 3: 프로세스가 중복 실행됨

**증상**: robot_webrtc.py가 여러 개 실행됨

**해결책**:
- Pi Gateway는 이미 실행 중인 프로세스를 감지하고 중복 실행을 방지합니다
- 수동으로 실행한 경우 먼저 종료:
  ```bash
  pkill -f robot_webrtc.py
  ```

## 요약

### 도커 실행 (권장)

| 컴포넌트 | 실행 방법 | 역할 |
|---------|----------|------|
| Pi Gateway | `docker-compose up` | 메인 서버 + WebRTC 시그널링 클라이언트 (자동) |
| MQTT 브릿지 | 로컬 실행 또는 별도 컨테이너 | MQTT 브릿지 |
| 고양이 감지 | Jetson에서 로컬 실행 | 고양이 감지 서비스 |

### 로컬 실행 구성 (3개 터미널)

| 터미널 | 실행 명령 | 역할 |
|--------|----------|------|
| 1 | `python3 -m src.main` | Pi Gateway 메인 서버 + WebRTC 시그널링 클라이언트 (자동) |
| 2 | `python3 scripts/mqtt_pi_bridge.py` | MQTT 브릿지 |
| 3 | `python3 scripts/cat_detection_service.py` | 고양이 감지 서비스 (Jetson) |

### WebRTC 스트리밍

- **자동 시작**: Pi Gateway가 프론트엔드 요청을 받아 자동으로 `robot_webrtc.py` 시작
- **별도 터미널 불필요**: 메인 서버에 통합되어 있음
- **수동 실행 가능**: 필요 시 `python3 scripts/robot_webrtc.py`로 별도 실행 가능

## 도커 vs 로컬 실행 비교

| 항목 | 도커 실행 | 로컬 실행 |
|------|----------|----------|
| **설정 복잡도** | 낮음 (docker-compose up) | 중간 (환경 변수 설정 필요) |
| **의존성 관리** | 자동 (Dockerfile) | 수동 (pip install) |
| **격리성** | 높음 (컨테이너 격리) | 낮음 (시스템과 공유) |
| **디버깅** | `docker logs` 사용 | 직접 터미널 출력 |
| **UART 접근** | `--device` 옵션 필요 | 직접 접근 가능 |
| **ROS2 통신** | `network_mode: host` 필요 | 직접 통신 가능 |

**권장**: 프로덕션 환경에서는 도커 실행을 권장합니다.
