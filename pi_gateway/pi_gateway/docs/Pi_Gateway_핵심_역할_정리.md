# Pi Gateway 핵심 역할 정리

## 개요

패트롤 팀이 경로 계획과 주행을 담당하더라도, Pi Gateway는 여전히 **필수적인 통신 브릿지** 역할을 합니다.

---

## 🎯 Pi Gateway의 핵심 역할

### 1. 웹 대시보드 ↔ ROS2 브릿지 ⭐ **핵심**

**역할**: 웹 대시보드의 명령을 ROS2 토픽으로 변환

**왜 필요한가?**
- 패트롤 팀은 자동 주행만 담당
- **사용자가 웹에서 수동 조작**할 때는 Pi Gateway가 처리해야 함
- 웹 대시보드와 ROS2는 서로 다른 프로토콜 사용

**동작**:
```
웹 대시보드 (WebSocket)
    ↓
Pi Gateway
    ├─ 수동 조작 명령 → /cmd_vel_teleop 발행
    ├─ 패트롤 명령 (/cmd_vel_nav) 구독
    └─ 통합 → /cmd_vel_out 발행
        ↓
UART 브릿지 → STM32
```

**없으면 안 되는 이유**:
- 웹 대시보드에서 로봇을 제어할 수 없음
- 수동 조작 모드가 작동하지 않음

---

### 2. 명령 통합 (Mux) ⭐ **핵심**

**역할**: 여러 입력 소스를 하나로 통합

**입력 소스**:
1. 웹 대시보드 명령 (`/cmd_vel_teleop`)
2. 패트롤 명령 (`/cmd_vel_nav`)

**통합 로직**:
- **teleop 모드**: 웹 대시보드 명령 우선
- **auto 모드**: 패트롤 명령 우선

**없으면 안 되는 이유**:
- 두 명령이 충돌할 수 있음
- 우선순위 관리 불가
- 안전 문제 발생 가능

---

### 3. 하드웨어 제어 (UART → STM32) ⭐ **핵심**

**역할**: ROS2 토픽을 UART 프레임으로 변환하여 STM32에 전송

**동작**:
```
ROS2 토픽 (/cmd_vel_out)
    ↓
Pi Gateway (RosCmdVelBridge)
    ├─ Twist 메시지 → UART 프레임 변환
    ├─ 바퀴 모터 제어
    ├─ 로봇팔 제어
    ├─ 급식 제어
    └─ 텔레메트리 수신
        ↓
STM32 (하드웨어)
```

**없으면 안 되는 이유**:
- ROS2 토픽은 소프트웨어 간 통신
- STM32는 UART 하드웨어 프로토콜 필요
- **패트롤 팀은 ROS2만 다룸, 하드웨어 제어는 Pi Gateway만 가능**

---

### 4. WebRTC 스트리밍 제어 ⭐ **핵심**

**역할**: 프론트엔드 요청을 받아 WebRTC 스트리밍 시작

**동작**:
```
프론트엔드 버튼 클릭
    ↓
STOMP 메시지 (/pub/robot/stream/start)
    ↓
Pi Gateway (webrtc_signaling_client.py)
    ├─ 메시지 수신
    └─ robot_webrtc.py 프로세스 시작
        ↓
WebRTC 스트리밍 시작
```

**없으면 안 되는 이유**:
- 프론트엔드와 로봇 간 통신 브릿지 필요
- 패트롤 팀은 주행만 담당, 스트리밍은 Pi Gateway 담당

---

### 5. MQTT 브릿지 ⭐ **중요**

**역할**: ROS2 토픽 ↔ MQTT 메시지 변환

**동작**:
```
ROS2 토픽
    ↓
Pi Gateway (mqtt_pi_bridge.py)
    ├─ ROS2 → MQTT 변환
    └─ MQTT → ROS2 변환
        ↓
백엔드 서버
```

**없으면 안 되는 이유**:
- 백엔드는 MQTT 사용
- ROS2와 MQTT는 다른 프로토콜
- 변환 브릿지 필요

---

### 6. 작업 완료 신호 발행 ⭐ **중요**

**역할**: 패트롤 노드가 필요로 하는 작업 완료 신호 발행

**토픽**:
- `/arm/done`: 로봇팔 작업 완료
- `/dock/done`: 도킹 완료
- `/servo/done`: 서보 작업 완료

**없으면 안 되는 이유**:
- 패트롤 노드는 작업 완료를 기다려야 함
- Pi Gateway가 하드웨어 제어를 담당하므로 완료 신호도 발행해야 함

## 왜 Pi Gateway가 필요한가?

### 1. 프로토콜 변환

**문제**: 각 컴포넌트가 다른 프로토콜 사용
- 웹 대시보드: WebSocket (FastAPI)
- 패트롤 노드: ROS2 토픽 (DDS)
- 백엔드: MQTT
- STM32: UART (프레임 프로토콜)

**해결**: Pi Gateway가 모든 프로토콜을 변환

### 2. 하드웨어 접근

**문제**: 패트롤 팀은 ROS2만 다룸
- STM32는 UART 하드웨어 프로토콜 필요
- ROS2 토픽을 UART 프레임으로 변환 필요

**해결**: Pi Gateway가 UART 통신 담당

### 3. 명령 통합

**문제**: 여러 입력 소스가 동시에 명령 발행 가능
- 웹 대시보드 (수동 조작)
- 패트롤 노드 (자동 주행)

**해결**: Pi Gateway가 우선순위 관리 및 통합

### 4. 실시간 통신

**문제**: 웹 대시보드와 로봇 간 실시간 통신 필요
- 텔레메트리 전송
- 제어 명령 수신
- WebRTC 스트리밍 제어

**해결**: Pi Gateway가 웹 대시보드 서버 운영

## 실제 동작 예시

### 시나리오 1: 사용자가 웹에서 수동 조작

```
사용자 (웹 대시보드)
    ↓ WebSocket
Pi Gateway
    ├─ 수동 조작 명령 수신
    ├─ /cmd_vel_teleop 발행
    ├─ CmdVelMux에서 통합
    └─ /cmd_vel_out 발행
        ↓
UART 브릿지 → STM32 → 바퀴 모터
```

**패트롤 팀**: 관여하지 않음 (수동 모드)

**Pi Gateway**: **필수** (웹 ↔ ROS2 브릿지)

---

### 시나리오 2: 패트롤 자동 주행

```
패트롤 노드
    ├─ 경로 계획
    ├─ /cmd_vel_nav 발행
    └─ 작업 요청 (/arm/cmd)
        ↓
Pi Gateway
    ├─ /cmd_vel_nav 구독
    ├─ /arm/cmd 구독 → Action 클라이언트 호출
    ├─ CmdVelMux에서 통합
    └─ /cmd_vel_out 발행
        ↓
UART 브릿지 → STM32 → 바퀴 모터
```

**패트롤 팀**: 경로 계획 및 주행 명령 발행

**Pi Gateway**: **필수** (명령 통합, 하드웨어 제어, 작업 완료 신호)

---

### 시나리오 3: 실시간 스트리밍

```
사용자 (웹 대시보드)
    ↓ 버튼 클릭
프론트엔드
    ↓ STOMP 메시지
Pi Gateway (webrtc_signaling_client.py)
    ├─ 메시지 수신
    └─ robot_webrtc.py 시작
        ↓
WebRTC 스트리밍
```

**패트롤 팀**: 관여하지 않음

**Pi Gateway**: **필수** (스트리밍 제어)

## 결론: Pi Gateway는 필수입니다

### 핵심 이유

1. **프로토콜 변환**: 웹 ↔ ROS2 ↔ MQTT ↔ UART
2. **하드웨어 제어**: STM32와 직접 통신 (UART)
3. **명령 통합**: 여러 입력 소스 통합 및 우선순위 관리
4. **웹 대시보드**: 사용자 인터페이스 제공
5. **실시간 통신**: 텔레메트리, 제어 명령, 스트리밍

### 비유

**패트롤 팀**: 자동차의 네비게이션 시스템 (경로 안내)
**Pi Gateway**: 자동차의 엔진과 조향장치 (실제 구동)

네비게이션이 경로를 안내해도, 엔진과 조향장치가 없으면 차가 움직이지 않습니다.

|------|----------|------------|

| 자동 주행 명령 | ✅ 발행 | ✅ 구독 및 통합 |
| 수동 조작 | ❌ 불필요 | ✅ **담당** |
| 하드웨어 제어 | ❌ 불가능 | ✅ **담당** |
| 웹 대시보드 | ❌ 불필요 | ✅ **담당** |
| WebRTC 스트리밍 | ❌ 불필요 | ✅ **담당** |
| MQTT 브릿지 | ❌ 불필요 | ✅ **담당** |

**결론**: 패트롤 팀이 주행을 담당하더라도, Pi Gateway는 **필수적인 통신 브릿지** 역할을 합니다.
