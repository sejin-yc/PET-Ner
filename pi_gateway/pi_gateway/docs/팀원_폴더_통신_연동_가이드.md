## ROS2 토픽 통신 분석

### 카메라 노드 (`camera_node.py`)

**발행 토픽**:
- `/camera/image_raw` (sensor_msgs/Image)
  - UDP 스트림에서 받은 카메라 영상 발행
  - UDP: `udp://192.168.100.254:5000`

**Pi Gateway 연동**:
- Pi Gateway는 이 토픽을 구독할 필요 없음
- 대신 `/front_cam/compressed` 토픽을 사용 (다른 노드가 발행)

---

### 아루코 마커 노드 (`aruco_pose_node.py`)

**구독 토픽**:
- `/front_cam/compressed` (sensor_msgs/CompressedImage)
  - 카메라 압축 영상 구독

**발행 토픽**:
- `/aruco/pose` (geometry_msgs/PoseStamped)
  - 아루코 마커의 위치/자세 정보
- `/aruco/id` (std_msgs/Int32)
  - 감지된 아루코 마커 ID
- `/image_aruco/compressed` (sensor_msgs/CompressedImage)
  - 디버그용 이미지 (마커 표시)

**Pi Gateway 연동**:
- Pi Gateway는 `/aruco/pose` 토픽을 구독하여 아루코 마커 위치 정보 활용 가능
- 현재는 사용하지 않지만, 향후 연동 시 참고

---

### 패트롤 노드 (`patrol.py` / `primitive_mission_manager_fixed_pwm.py`)

**구독 토픽**:
- TF (Transform): `map` → `base_link`
  - 로봇의 현재 위치/자세 정보
- `/dock/done` (std_msgs/Bool) - 도킹 완료 신호
- `/arm/done` (std_msgs/Bool) - 로봇팔 작업 완료 신호
- `/servo/done` (std_msgs/Bool) - 서보 작업 완료 신호

**발행 토픽**:
- `/cmd_vel_nav` (geometry_msgs/Twist)
  - Nav2/패트롤 명령 (자동 주행)
- `/gateway/allow_strafe` (std_msgs/Bool)
  - 횡이동 허용 여부

**Pi Gateway 연동**:
- ✅ **Pi Gateway는 `/cmd_vel_nav` 토픽을 구독해야 함**
- ✅ **Pi Gateway는 `/gateway/allow_strafe` 토픽을 구독해야 함**
- ✅ **Pi Gateway는 작업 완료 신호를 발행해야 함**:
  - `/dock/done` (도킹 완료)
  - `/arm/done` (로봇팔 작업 완료)
  - `/servo/done` (서보 작업 완료)

---

### UART 브릿지 노드 (`twist_to_stm_uart_bridge.py`)

**구독 토픽**:
- `/cmd_vel` (geometry_msgs/Twist)
  - 최종 속도 명령 (twist_mux 출력)

**발행 토픽**: 없음

**동작**:
- `/cmd_vel` 토픽을 구독하여 STM32로 UART 전송
- 고정 PWM 방식 사용

**Pi Gateway 연동**:
- ⚠️ **충돌 가능성**: Pi Gateway도 `/cmd_vel`을 발행함
- ✅ **해결책**: Pi Gateway는 `/cmd_vel_out` 또는 다른 토픽 사용
- 또는 twist_mux를 통해 통합

## Pi Gateway 통신 요구사항

### 1. ROS2 토픽 발행 (Pi Gateway → 다른 노드)

#### 필수 발행 토픽

| 토픽명 | 메시지 타입 | 용도 | 수신자 |
|--------|------------|------|--------|
| `/cmd_vel_out` | geometry_msgs/Twist | 최종 속도 명령 | twist_mux 또는 UART 브릿지 |
| `/dock/done` | std_msgs/Bool | 도킹 완료 신호 | 패트롤 노드 |
| `/arm/done` | std_msgs/Bool | 로봇팔 작업 완료 신호 | 패트롤 노드 |
| `/servo/done` | std_msgs/Bool | 서보 작업 완료 신호 | 패트롤 노드 |

#### 선택 발행 토픽

| 토픽명 | 메시지 타입 | 용도 | 수신자 |
|--------|------------|------|--------|
| `/control_mode` | std_msgs/String | 제어 모드 ("teleop" / "auto") | 다른 노드들 |
| `/gateway/allow_strafe` | std_msgs/Bool | 횡이동 허용 여부 | 패트롤 노드 (이미 구독 중) |

#### 선택 구독 토픽

| 토픽명 | 메시지 타입 | 발행자 | 용도 |

| `/aruco/id` | std_msgs/Int32 | 아루코 마커 노드 | 감지된 마커 ID |

### 3. 웹 대시보드 통신 (Pi Gateway ↔ FE/BE)

#### HTTP REST API (Pi Gateway → 백엔드)

| 엔드포인트 | 메서드 | 용도 |
|-----------|--------|------|
| `POST /camera/stream/start` | POST | WebRTC 스트리밍 시작 (선택사항) |
| `POST /camera/stream/stop` | POST | WebRTC 스트리밍 중지 (선택사항) |
| `GET /camera/stream/status` | GET | 스트리밍 상태 확인 (선택사항) |

#### WebSocket (Pi Gateway ↔ 프론트엔드)

| 프로토콜 | 용도 |
|---------|------|
| WebSocket (FastAPI) | 실시간 제어 명령 수신, 텔레메트리 전송 |

#### STOMP over WebSocket (Pi Gateway ↔ 시그널링 서버)

| 토픽 | 방향 | 용도 |
|------|------|------|
| `/sub/robot/stream/start` | 구독 | 프론트엔드 스트리밍 시작 요청 수신 |
| `/pub/peer/offer` | 발행 | WebRTC Offer 발행 (robot_webrtc.py) |
| `/sub/peer/answer` | 구독 | WebRTC Answer 수신 (robot_webrtc.py) |

| 토픽 | 방향 | 용도 |

## 토픽 충돌 해결 방안

### 문제: `/cmd_vel` 토픽 충돌

**현황**:
- Pi Gateway: `/cmd_vel` 발행 (웹 대시보드 명령)
- 팀원 UART 브릿지: `/cmd_vel` 구독 (STM32 전송)

**해결 방안**:

#### 방안 1: twist_mux 사용 (권장)

```
Pi Gateway → /cmd_vel_teleop
패트롤 노드 → /cmd_vel_nav
         ↓
    twist_mux
         ↓
    /cmd_vel (최종 명령)
         ↓
UART 브릿지 → STM32
```

**장점**: 
- 표준 ROS2 패턴
- 우선순위 관리 가능
- 충돌 없음

#### 방안 2: 토픽명 변경

Pi Gateway가 `/cmd_vel_out` 발행:
```
Pi Gateway → /cmd_vel_out
패트롤 노드 → /cmd_vel_nav
         ↓
    통합 노드 (Pi Gateway 내부)
         ↓
    /cmd_vel (최종 명령)
         ↓
UART 브릿지 → STM32
```

**장점**: 
- 간단한 구현
- Pi Gateway 내부에서 통합 가능

## 로봇팔 통신

### robotic_arm 폴더 분석

**역할**: LeRobot 기반 로봇팔 제어

**통신 방식**:
- 직접 하드웨어 제어 (UART: `/dev/ttyACM0`)
- ROS2 토픽 사용 안 함

**Pi Gateway 연동**:
- 현재 Pi Gateway의 `ArmControlBridge`가 로봇팔 제어 담당
- UART를 통해 STM32로 명령 전송
- 로봇팔 작업 완료 시 `/arm/done` 토픽 발행 필요

## 요약

### Pi Gateway의 역할

1. **웹 대시보드 ↔ ROS2 브릿지**
   - 웹 명령 → ROS2 토픽 변환
   - ROS2 텔레메트리 → 웹 전송

2. **ROS2 토픽 통합**
   - `/cmd_vel_nav` (패트롤) + `/cmd_vel_teleop` (웹) → `/cmd_vel` (최종)
   - 작업 완료 신호 발행 (`/arm/done`, `/dock/done` 등)

3. **하드웨어 제어**
   - UART를 통한 STM32 제어
   - 로봇팔, 급식, 바퀴 모터 제어

4. **WebRTC 스트리밍 제어**
   - 프론트엔드 요청 수신 → `robot_webrtc.py` 시작

### 통신 프로토콜

| 통신 대상 | 프로토콜 | 용도 |
|----------|---------|------|
| 프론트엔드 | WebSocket (FastAPI) | 실시간 제어/텔레메트리 |
| 백엔드 | HTTP REST API, MQTT | 데이터 전송 |
| 시그널링 서버 | STOMP over WebSocket | WebRTC 시그널링 |
| ROS2 노드들 | ROS2 토픽 (DDS) | 자동 주행, 센서 데이터 |
| STM32 | UART | 하드웨어 제어 |
