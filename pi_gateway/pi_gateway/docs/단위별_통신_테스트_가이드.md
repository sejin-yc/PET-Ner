# 단위별 통신 테스트 가이드

나중에 **내 코드(Pi Gateway)** 로 통신 테스트할 때, **구간마다** 값을 확인하는 방법입니다.

---

## 1. 전체 흐름 (한눈에)

```
[입력]
  /cmd_vel (Nav2)  또는  cmd_vel_teleop (웹)
       │
       ▼
  control_mode (auto / teleop)
       │
       ▼
  CmdVelMux  →  cmd_vel_out (선택된 하나)
       │
       ▼
  RosCmdVelBridge  →  UART TX (aa55010c ...)
       │
       ▼
  STM32 (모터)
```

**단위별로**: ① 입력 → ② mode → ③ Mux 출력 → ④ UART TX → (⑤ UART RX) 순서로 확인하면 됩니다.

---

## 2. 단위 ①: 입력 (Nav2 / 웹)

### 2-1. Nav2가 보내는지 (`/cmd_vel`)

Pi Gateway가 떠 있는 **같은 ROS 도메인**에서:

```bash
# 토픽 목록
ros2 topic list | grep cmd_vel

# 값 한 번만 보기 (자율 주행 중이면 0이 아닌 값 나와야 함)
ros2 topic echo /cmd_vel --once
```

**기대**: `linear.x`, `angular.z` 등 0이 아닌 값이 주기적으로 나오면 Nav2 입력 정상.

### 2-2. 웹(조이스틱)이 보내는지 (`cmd_vel_teleop`)

```bash
ros2 topic echo /cmd_vel_teleop --once
```

웹에서 조이스틱 움직인 뒤 실행하면 0이 아닌 값이 나와야 함.

---

## 3. 단위 ②: control_mode

```bash
ros2 topic echo /control_mode --once
```

**기대**: `data: "auto"` 또는 `data: "teleop"`.

- **auto** → Mux는 `/cmd_vel` 사용
- **teleop** → Mux는 `cmd_vel_teleop` 사용

---

## 4. 단위 ③: Mux 출력 (`cmd_vel_out`)

Mux가 **선택한** 최종 속도입니다. 여기까지 오면 UART 직전 단계입니다.

```bash
ros2 topic echo /cmd_vel_out --once
```

**기대**:
- 자율 주행 중: Nav2와 같은 값
- 수동 조종 중: 조이스틱 값
- 둘 다 없으면: `linear.x: 0.0`, `angular.z: 0.0` (정지)

**여기서 계속 0이면** → 입력(/cmd_vel 또는 cmd_vel_teleop) 또는 control_mode 확인.

---

## 5. 단위 ④: UART TX (Pi → STM32)

Pi Gateway(main.py) 실행 시 **환경 변수**로 TX를 단위별로 볼 수 있습니다.

| 환경 변수 | 의미 | 사용 예 |
|-----------|------|----------|
| `LOG_LEVEL=DEBUG` | DEBUG 로그 출력 | 모든 단계 |
| `UART_DEBUG_TX=1` | **보내는 프레임** hex 출력 | ④ UART TX |
| `UART_DEBUG_RX=1` | **받는 프레임** hex 출력 | ⑤ UART RX |
| `UART_LINK_STATS=1` | 10초마다 TX/RX 개수만 한 줄 | 부하 적게 보기 |

### 도커 예시 (TX만 보고 싶을 때)

```bash
-e LOG_LEVEL=DEBUG \
-e UART_DEBUG_TX=1 \
```

### 해석

| TX hex | 의미 |
|--------|------|
| `aa55020002` | Heartbeat (ID 0x02) |
| `aa55010c0000000000000000000000000d` | cmd_vel (ID 0x01) **vx=0, vy=0, wz=0** → 정지 |
| `aa55010c...` (중간에 0이 아닌 float) | cmd_vel **움직임** |

④ 단위 테스트: **조이스틱이나 Nav2로 움직일 때** 위 cmd_vel 프레임이 0이 아닌 값으로 바뀌는지 보면 됩니다.

---

## 6. 단위 ⑤: UART RX (STM32 → Pi) – 선택

텔레메트리(배터리, 엔코더 등)가 들어오는지 확인할 때:

```bash
-e LOG_LEVEL=DEBUG \
-e UART_DEBUG_RX=1 \
```

RX 로그에 `ID_BATTERY(0x81)`, `ID_ENCODER(0x82)` 등이 주기적으로 나오면 STM32 → Pi 수신 정상.

---

## 7. HTTP API로 확인 (ROS 없이)

Pi Gateway가 떠 있으면 **HTTP**로도 일부 단위 확인 가능합니다.

| 확인 항목 | 명령 |
|-----------|------|
| 현재 상태(mode, estop 등) | `curl http://localhost:8000/debug/state` |
| 로봇 헬스(모드, 배터리 등) | `curl http://localhost:8000/robot/health` |
| 텔레메트리 최신값 | `curl http://localhost:8000/telemetry/latest` |

---

## 8. 체크리스트 (내 코드로 통신 테스트할 때)

| 순서 | 단위 | 확인 방법 | 통과 조건 |
|------|------|------------|-----------|
| 1 | 입력 | `ros2 topic echo /cmd_vel --once` 또는 `cmd_vel_teleop` | 자율/수동 시 0이 아닌 값 |
| 2 | mode | `ros2 topic echo /control_mode --once` | auto 또는 teleop |
| 3 | Mux 출력 | `ros2 topic echo /cmd_vel_out --once` | 움직일 때 0이 아님 |
| 4 | UART TX | `UART_DEBUG_TX=1` + 로그 | cmd_vel hex에 0이 아닌 float |
| 5 | UART RX | `UART_DEBUG_RX=1` (선택) | 배터리/엔코더 등 수신 |

---

## 9. 요약

- **단위별로**: 입력 → control_mode → cmd_vel_out → UART TX (→ UART RX) 순서로 확인.
- **ROS**: `ros2 topic echo /cmd_vel`, `/control_mode`, `/cmd_vel_out` 로 구간마다 값 확인.
- **UART**: `UART_DEBUG_TX=1`, `UART_DEBUG_RX=1`, `LOG_LEVEL=DEBUG` 로 프레임 단위 확인.
- **HTTP**: `curl .../debug/state`, `/robot/health`, `/telemetry/latest` 로 상태/텔레메트리 확인.

이렇게 하면 나중에 내 코드로 통신 테스트할 때 **어느 구간에서 막히는지** 바로 짚을 수 있습니다.
